# Chapter 56: Testing Angular Applications

## 1. Overview

Testing an Angular application is not one activity — it is a layered discipline. At the bottom sit fast, isolated unit tests that verify a single class's logic. Above that sit component tests that exercise templates, bindings, and change detection through `TestBed`. Above that sit integration tests that wire several components/services together, still inside the same process. At the top sit end-to-end (E2E) tests that drive a real browser against a running application.

Angular's testing story has evolved considerably:

- **Historically**: Jasmine (assertion/spy framework) + Karma (browser test runner) generated by the Angular CLI, running inside a real (or headless) browser via Zone.js-patched change detection.
- **Modern (Angular 15+/16+/17+)**: first-class support for **Jest** (faster, runs in Node with jsdom, better watch mode, snapshot testing) and the **Web Test Runner** / `@web/test-runner` based `@angular/build:karma` replacement, plus experimental **zoneless** change detection which changes how `detectChanges()` behaves.
- E2E tooling moved from **Protractor** (deprecated, removed from CLI schematics in Angular 12+) to **Cypress** and **Playwright**, both now offered as official `ng add`-able builders.

A staff-level engineer is expected to reason about *why* a given test is written at a given layer (test pyramid), to configure `TestBed` correctly for standalone components and OnPush strategies, to test reactive primitives (RxJS via marble testing, Signals via `effect`/`computed`), to mock HTTP deterministically, and to pick the right harness abstraction (`ComponentHarness`) so tests survive DOM refactors.

## 2. Core Concepts

### 2.1 The Test Pyramid for Angular

```
        ▲  E2E (Cypress / Playwright)      – few, slow, high confidence, real browser+backend
       ▲▲▲ Integration (TestBed, multiple  – some, medium speed, real DI graph
      ▲▲▲▲▲ collaborators, HttpTestingController)
     ▲▲▲▲▲▲▲ Unit (services, pipes, pure   – many, fast, isolated
    ▲▲▲▲▲▲▲▲▲ functions, component classes)
```

- **Unit tests**: test a class in isolation — a service's business logic, a pipe's `transform`, a component's *class* (not template) by `new`-ing it directly when it has no DI, or via `TestBed` with all dependencies mocked.
- **Component/integration tests**: use `TestBed` to compile the component + template, verify DOM output, event bindings, child component interaction (shallow vs deep — see below), and interaction with real (or faked) services.
- **E2E tests**: run the compiled app in a real browser with Cypress/Playwright, hitting a real or mocked backend, verifying user journeys (login → navigate → submit form → see result).

The pyramid shape is a **cost/confidence trade-off**: unit tests are cheap to write and run (milliseconds), but a green unit-test suite says nothing about whether components actually render correctly together or whether routing/guards work. E2E tests give the highest confidence but are slow, flaky, and expensive to maintain — hence "few at the top."

### 2.2 Jasmine/Karma vs Jest vs Web Test Runner

| Aspect | Jasmine + Karma (classic CLI default) | Jest | Web Test Runner (`@angular/build:karma` successor) |
|---|---|---|---|
| Execution environment | Real/headless browser (Chrome via Karma launcher) | Node.js + jsdom (DOM emulation) | Real browser via Playwright/Puppeteer providers |
| Speed | Slower (browser bootstrap per run) | Fast (no browser boot, parallel workers) | Fast, but still real browser semantics |
| Watch mode | Basic | Excellent, smart re-run of affected tests | Good |
| Snapshot testing | No (Jasmine has no snapshot support) | Yes | No |
| DOM fidelity | Perfect (real browser) | Approximate (jsdom gaps: layout, some CSSOM, `IntersectionObserver` etc. need polyfills) | Perfect (real browser) |
| Angular CLI support | `ng test` default historically | `ng add @angular-builders/jest` (community) or native experimental Jest builder (`@angular/build:unit-test` circa Angular 20) | Native builder replacing Karma (Karma deprecated & removed as CLI default starting Angular 17+) |
| Assertion/mocking API | `expect(...).toBe(...)`, `jasmine.createSpy` | `expect(...).toBe(...)`, `jest.fn()` (similar API, subtle differences e.g. `toHaveBeenCalledWith` matcher semantics) | Same as whichever framework is layered on top (often Jasmine or Jest-compatible via `@web/test-runner-jasmine`) |

Key point for interviews: **Karma is deprecated** as the Angular team's own test runner (announced deprecation in Angular 17, `ng new` no longer scaffolds Karma by default in recent versions) in favor of build-tool-native runners (esbuild/Vite based) with either a browser (Web Test Runner) or Node (Jest) execution model. Jasmine as the *assertion syntax* often persists even when Karma-the-runner is swapped out, because most Angular ecosystem test helpers (`TestBed`, `ComponentFixture`) are runner-agnostic and only need a global `describe/it/expect`.

### 2.3 TestBed Fundamentals

`TestBed` is Angular's test-time dependency injection & compilation harness. It:

1. Creates a **dynamic testing module** (historically an `NgModule`; for standalone components/directives/pipes it creates an equivalent standalone bootstrapping context) that declares/imports the thing under test plus any stated dependencies.
2. Compiles the component's template (`TestBed.compileComponents()` — mostly a no-op with the Ivy JIT-less AOT test pipeline now, but still needed when templates are fetched via `templateUrl` in certain configurations, or for older/legacy configurations).
3. Produces a `ComponentFixture<T>`, which wraps the created component instance (`fixture.componentInstance`), its native/debug DOM node (`fixture.nativeElement` / `fixture.debugElement`), and control over change detection (`fixture.detectChanges()`).

```typescript
TestBed.configureTestingModule({
  imports: [MyStandaloneComponent], // standalone components go in `imports`, not `declarations`
  providers: [
    { provide: DataService, useValue: fakeDataService }
  ]
});
```

For **standalone components**, there is no `declarations` array — a standalone component/directive/pipe is imported like a module. If the component under test itself imports other standalone components/directives it needs (e.g. `CommonModule`, child standalone components), those come along automatically since Angular resolves the standalone component's own `imports` metadata; you do not need to re-declare them in `TestBed`, only override/replace them if you want to stub a child.

### 2.4 Shallow vs Deep Component Testing

- **Deep (integration-style) testing**: import the real child components as declared by the parent. This verifies the whole rendered tree, including child template output, but is slower, more brittle (a child's internal change can break the parent's test), and requires all of the child's own dependencies to be satisfiable.
- **Shallow testing**: replace child components with stub/mock versions (e.g. using `NO_ERRORS_SCHEMA`, `CUSTOM_ELEMENTS_SCHEMA`, or explicit fake components with the same selector) so the test only cares about the parent's own logic/template bindings, ignoring child internals.

```typescript
TestBed.configureTestingModule({
  imports: [ParentComponent],
  schemas: [NO_ERRORS_SCHEMA] // unknown child elements/attributes are ignored, not errored
});
```

`NO_ERRORS_SCHEMA` is a blunt instrument — it silently swallows binding typos on **any** unknown element/attribute, so many teams prefer creating explicit stub components (`@Component({ selector: 'app-child', template: '' })`) registered in place of the real one, which still validates that the selector/inputs/outputs are wired correctly while skipping the child's real rendering.

### 2.5 Testing OnPush Components

`ChangeDetectionStrategy.OnPush` components only re-render when: an `@Input()` reference changes, an event originating from the component (or its children) fires, an `async` pipe emits, a signal read in the template changes, or `markForCheck()`/`detectChanges()` is called explicitly.

In tests, **`fixture.detectChanges()` is not automatically re-invoked** after you mutate component state — you must call it again after every meaningful mutation, even for `Default` strategy components, but it is especially easy to forget with `OnPush` because there's an intuition that "OnPush needs something extra" — in fact the API is the same; the difference is only that changing a nested/mutable object *without* replacing the reference will not trigger CD even if you call `detectChanges()`, because Angular's OnPush check still ultimately relies on `Input` identity or explicit dirty-marking, not `detectChanges()` bypassing it. `fixture.detectChanges()` **forces** a check of that component regardless of strategy, so OnPush components in tests generally behave "normally" as long as you call it after each mutation — the classic gotcha is mutating an `@Input`-bound object in place (not replacing the reference) and expecting the *component itself*, when driven by a real OnPush-respecting parent, to pick it up; when the component is the *root* of the fixture, `detectChanges()` bypasses that nuance since it's an explicit imperative check.

### 2.6 Testing Signals, Computed, and Effects

- **Signals** (`signal()`) are plain synchronous reads/writes — no async machinery needed in tests: call `.set()`/`.update()` and assert `.()` reads or assert the rendered DOM after `fixture.detectChanges()`.
- **`computed()`** values are lazily recomputed on read; testing them is just reading `.()` after changing upstream signals — no scheduler tricks needed since computed signals are pull-based/memoized, not scheduled.
- **`effect()`** runs asynchronously by default, scheduled by Angular's reactivity scheduler (an internal microtask-based scheduler independent of Zone.js). This means in a test, an `effect()` registered via `TestBed.runInInjectionContext()` or inside a component will **not** run synchronously after a signal write — you need to either:
  - call `fixture.detectChanges()` (which flushes pending effects as part of the CD cycle in component contexts), or
  - use `TestBed.tick()` (Angular 17+) or await a microtask (`await Promise.resolve()`), or
  - for injector-level effects created via `TestBed.runInInjectionContext(() => effect(...))`, call `TestBed.flushEffects()`-equivalent behavior — in practice most guidance is to wrap in `fixture.whenStable()`/`await fixture.whenRenderingDone()` or explicitly call `TestBed.tick()`.

### 2.7 Marble Testing RxJS with TestScheduler

RxJS ships `TestScheduler` for **virtual time** testing: marble diagrams encode emissions, time gaps, completions, and errors as ASCII strings, letting you assert async operator chains (like `debounceTime`, `switchMap`, `retry`) synchronously and deterministically — no real `setTimeout`/`fakeAsync` flushing needed for the RxJS layer itself.

Marble syntax:

- `-` : one frame (virtual millisecond, but conventionally each `-`/character = 10 "frames" e.g. depends on configured frame time; default is 1 character = 1 virtual frame unit, scaled by `run()`'s internal timing — practically treat each dash as a tick of the configured time unit).
- `a`, `b`, `c` : emitted **next** values, mapped through a `values` object to real values.
- `|` : completion.
- `#` : error.
- `()` : groups multiple events into the same virtual frame (e.g. `(ab|)` = `a` and `b` and complete, all synchronously in the same frame).
- `^` : subscription point marker (for "hot" observables / expected subscription diagrams).

```typescript
testScheduler.run(({ cold, hot, expectObservable, expectSubscriptions }) => {
  const source$ = cold('-a-b-c|');
  const expected =      '-a-b-c|';
  expectObservable(source$).toBe(expected);
});
```

### 2.8 Mocking HTTP with HttpTestingController

`HttpClientTestingModule` (or, in modern standalone setups, `provideHttpClientTesting()` alongside `provideHttpClient()`) replaces the real `HttpBackend` with a controllable stub. Tests:

1. Trigger the code that calls `HttpClient`.
2. Call `httpMock.expectOne(url)` to assert exactly one matching request was made, and get a `TestRequest` handle.
3. Call `req.flush(mockData)` (or `req.error(...)`, `req.flush(null, { status: 500, statusText: 'Server Error' })`) to synchronously resolve the request.
4. Call `httpMock.verify()` (usually in `afterEach`) to assert no unmatched/outstanding requests remain — this catches accidental extra calls or unflushed requests that would otherwise silently pass.

### 2.9 CDK Component Test Harnesses

Angular Material/CDK ship a `ComponentHarness` abstraction (`@angular/cdk/testing`) that decouples tests from DOM structure. Instead of querying `.mat-mdc-button` classes directly (which are internal implementation details that can change between Material versions), you load a harness (`MatButtonHarness`, or a custom harness for your own components) via a `HarnessLoader`, and call semantic methods (`.click()`, `.getText()`, `.isDisabled()`). This works identically whether the test runs synchronously (`TestbedHarnessEnvironment`) or via Protractor/E2E-style harness environments, because harnesses abstract "how to interact" from "how to locate."

```typescript
const loader = TestbedHarnessEnvironment.loader(fixture);
const button = await loader.getHarness(MatButtonHarness.with({ text: 'Submit' }));
await button.click();
```

Writing **custom harnesses** for your own reusable components is a staff-level practice: it lets consuming teams' tests remain stable across your component's internal markup refactors.

### 2.10 E2E: Cypress vs Playwright

| | Cypress | Playwright |
|---|---|---|
| Architecture | Runs inside the browser (same event loop as app under some proxy tricks), Node process manages orchestration | Drives real browsers via CDP/WebSocket from an external Node process; supports Chromium, Firefox, WebKit |
| Cross-browser | Historically Chromium-family + limited Firefox/WebKit (experimental) | First-class multi-browser + multi-tab, mobile emulation |
| Speed/parallelism | Good, built-in dashboard for parallelization (paid tier for full features) | Very fast, built-in test runner with free parallel workers/sharding |
| Auto-waiting | Yes (retries assertions) | Yes (actionability checks before every action) |
| Network stubbing | `cy.intercept()` | `page.route()` |
| Angular CLI integration | `ng add @cypress/schematic` (or the now-community-run Cypress builder) | `ng e2e` can be configured to use `@angular/build`'s Playwright builder / `ng add playwright` community schematics |
| Debugging | Time-travel snapshots in the Cypress runner UI | Trace viewer, video, step-by-step trace with DOM snapshots |

Both replaced **Protractor**, which was tightly coupled to Angular's (deprecated) `angular.getTestability()` API and WebDriver, and was formally deprecated/removed from the CLI.

## 3. Code Examples

### 3.1 Standalone Component TestBed Setup

```typescript
import { Component, Input, signal } from '@angular/core';
import { TestBed } from '@angular/core/testing';
import { By } from '@angular/platform-browser';

@Component({
  selector: 'app-greeting',
  standalone: true,
  template: `<h1>Hello, {{ name }}!</h1>`
})
class GreetingComponent {
  @Input() name = 'World';
}

describe('GreetingComponent (standalone)', () => {
  beforeEach(async () => {
    await TestBed.configureTestingModule({
      imports: [GreetingComponent] // standalone component imported directly, no declarations[]
    }).compileComponents();
  });

  it('renders the provided name', () => {
    const fixture = TestBed.createComponent(GreetingComponent);
    fixture.componentInstance.name = 'Angular';
    fixture.detectChanges(); // triggers initial CD; required before DOM reflects the Input

    const h1 = fixture.debugElement.query(By.css('h1'));
    expect(h1.nativeElement.textContent).toContain('Hello, Angular!');
  });
});
```

### 3.2 OnPush Component Needing Manual detectChanges()

```typescript
import { Component, Input, ChangeDetectionStrategy } from '@angular/core';
import { TestBed } from '@angular/core/testing';
import { By } from '@angular/platform-browser';

@Component({
  selector: 'app-counter-badge',
  standalone: true,
  changeDetection: ChangeDetectionStrategy.OnPush,
  template: `<span class="count">{{ count }}</span>`
})
class CounterBadgeComponent {
  @Input() count = 0;
}

describe('CounterBadgeComponent (OnPush)', () => {
  it('does NOT reflect a mutation until detectChanges() runs again', () => {
    const fixture = TestBed.createComponent(CounterBadgeComponent);
    fixture.detectChanges(); // initial render, count = 0

    fixture.componentInstance.count = 5;
    const span = fixture.debugElement.query(By.css('.count')).nativeElement;

    // Common false-negative trap: asserting BEFORE calling detectChanges() again.
    // At this point span.textContent is still "0" — the view has not been re-checked.
    expect(span.textContent).toBe('0');

    fixture.detectChanges(); // force a fresh CD pass — OnPush component is the fixture root,
                             // so detectChanges() checks it unconditionally.
    expect(span.textContent).toBe('5');
  });
});
```

### 3.3 Signal / Computed Test

```typescript
import { signal, computed } from '@angular/core';

describe('signals + computed', () => {
  it('recomputes derived value when the source signal changes', () => {
    const price = signal(100);
    const quantity = signal(2);
    const total = computed(() => price() * quantity());

    expect(total()).toBe(200);

    quantity.set(3);
    // computed() is pull-based and memoized: no scheduler flush needed,
    // the new value is available on the very next read.
    expect(total()).toBe(300);
  });
});
```

```typescript
import { Component, effect, signal } from '@angular/core';
import { TestBed } from '@angular/core/testing';

@Component({
  standalone: true,
  selector: 'app-logger',
  template: ``
})
class LoggerComponent {
  value = signal(0);
  log: number[] = [];
  constructor() {
    effect(() => { this.log.push(this.value()); });
  }
}

describe('effect() timing', () => {
  it('runs the effect only after a change-detection flush', () => {

    fixture.detectChanges(); // flushes the initial effect scheduling -> log: [0]

    expect(fixture.componentInstance.log).toEqual([0, 1]);
  });
});
```

### 3.4 Marble Test: Debounced Search

```typescript
import { TestScheduler } from 'rxjs/testing';
import { debounceTime, distinctUntilChanged, switchMap } from 'rxjs/operators';
import { of, Subject } from 'rxjs';

function search(query: string) {
  return of(`result-for-${query}`);
}

describe('debounced search stream', () => {
  let scheduler: TestScheduler;

  beforeEach(() => {
    scheduler = new TestScheduler((actual, expected) => {
      expect(actual).toEqual(expected);
    });
  });

  it('debounces rapid keystrokes and only searches for the settled value', () => {
    scheduler.run(({ cold, expectObservable }) => {
      // 'a' at frame 0, 'b' at frame 2 (still typing), 'c' at frame 4 then a pause -> settles
      const input$ = cold('a-b-c-----|', {
        a: 'A', b: 'AB', c: 'ABC'
      });

      const result$ = input$.pipe(
        debounceTime(5, scheduler),
        distinctUntilChanged(),
        switchMap(query => search(query))
      );

      // Only the final settled term ('ABC') survives the 5-frame debounce window;
      // intermediate keystrokes 'A' and 'AB' are dropped since they arrive
      // before the debounce window elapses.
      const expected = '------(c|)';
      expectObservable(result$).toBe(expected, { c: 'result-for-ABC' });
    });
  });
});
```

### 3.5 HttpTestingController Test

```typescript
import { TestBed } from '@angular/core/testing';
import { provideHttpClient } from '@angular/common/http';
import { provideHttpClientTesting, HttpTestingController } from '@angular/common/http/testing';
import { HttpClient } from '@angular/common/http';
import { firstValueFrom } from 'rxjs';

interface User { id: number; name: string; }

class UserService {
  constructor(private http: HttpClient) {}
  getUser(id: number) {
    return this.http.get<User>(`/api/users/${id}`);
  }
}

describe('UserService', () => {
  let service: UserService;
  let httpMock: HttpTestingController;

  beforeEach(() => {
    TestBed.configureTestingModule({
      providers: [
        UserService,
        provideHttpClient(),
        provideHttpClientTesting()
      ]
    });
    service = TestBed.inject(UserService);
    httpMock = TestBed.inject(HttpTestingController);
  });

  afterEach(() => {
    httpMock.verify(); // fails the test if there are unmatched/outstanding requests
  });

  it('fetches a user by id', async () => {
    const promise = firstValueFrom(service.getUser(42));

    const req = httpMock.expectOne('/api/users/42');
    expect(req.request.method).toBe('GET');

    req.flush({ id: 42, name: 'Ada Lovelace' });

    const user = await promise;
    expect(user.name).toBe('Ada Lovelace');
  });

  it('propagates a server error', async () => {
    const promise = firstValueFrom(service.getUser(99));
    const req = httpMock.expectOne('/api/users/99');

    req.flush('Not Found', { status: 404, statusText: 'Not Found' });

    await expectAsync(promise).toBeRejected();
  });
});
```

## 4. Internal Working

### 4.1 How TestBed Creates an Isolated Environment Per Test

- `TestBed` maintains a module-resolution cache keyed by the testing module's metadata. Calling `TestBed.configureTestingModule({...})` records the desired imports/providers/declarations/schemas but does not compile anything yet.
- On the first `TestBed.createComponent()` (or an explicit `compileComponents()`), Angular's Ivy compiler generates (or reuses, if AOT-precompiled) the component's rendering instructions, and `TestBed` instantiates a **fresh dependency injection graph**: an environment injector (module-level, akin to what `createNgModuleRef`/`createEnvironmentInjector` build for the app root) layered under the platform injector, plus a fresh element injector chain per created component tree. This mirrors production bootstrap (`bootstrapApplication`/`platformBrowserDynamic().bootstrapModule()`) but scoped to the test file/spec.
- Each call to `TestBed.resetTestingModule()` (automatically invoked between tests by the Angular testing framework integration, e.g. via the Jasmine/Jest environment hook) tears down that injector and its instances, so tests do not leak singleton service state into one another — this is why a `providedIn: 'root'` singleton service gets a **brand-new instance per test**, not the same instance reused across the suite.
- `ComponentFixture` wraps the created `ComponentRef`; `fixture.detectChanges()` calls into the same `ApplicationRef`-style tick logic that production CD uses, but invoked manually and synchronously by the test rather than being scheduled implicitly by Zone.js's `onMicrotaskEmpty`/coalesced scheduling. In production, Zone.js patches async APIs (`setTimeout`, promises, DOM events) and Angular's `NgZone` triggers `tick()` automatically whenever the zone becomes stable; in tests, Zone.js's monkey-patches are still present (unless running zoneless), but Angular deliberately does **not** auto-tick on every microtask inside a synchronous unit test the way it does in a running app, because a real app has an `ApplicationRef.tick()` subscription hooked to `NgZone.onMicrotaskEmpty`, while `TestBed`'s fixture is deliberately left in manual mode by default so assertions can happen deterministically between renders. (`fixture.autoDetectChanges(true)` opts back into automatic re-checking via the same zone-stable hook, at the cost of losing the ability to inspect "pre-render" DOM state.)

### 4.2 Why fixture.detectChanges() Must Be Manual

Zone.js in a browser app hooks `NgZone.run()` around event handlers/timers/XHR callbacks so that after any async callback completes, if the zone is stable, `ApplicationRef.tick()` fires, walking every root component's view and running change detection top-down. In a **test**, if this happened automatically after every synchronous statement, assertions about intermediate states (e.g., "before requestion resolves, a loading spinner is shown") would be impossible to write reliably, since the DOM would already reflect the final state by the time your `expect()` runs. `TestBed`'s default (`fixture.autoDetectChanges(false)`, the default) hands control back to the test author: you decide exactly when a CD pass happens, which is essential for testing loading states, debounce boundaries, and OnPush inputs precisely.

Under **zoneless** change detection (`provideExperimentalZonelessChangeDetection()` / stabilized `provideZonelessChangeDetection()` in newer Angulars), there is no Zone.js patching at all; change detection is instead driven by signals' reactive graph notifying `ApplicationRef` directly, plus explicit scheduling via a microtask-coalesced scheduler. `fixture.detectChanges()` remains the same public API for tests, but internally it now triggers the zoneless scheduler's flush rather than relying on `NgZone`; this is largely transparent to test authors as long as they don't depend on Zone-specific timing APIs like `fakeAsync`'s zone-patched `tick()` for scheduling (zoneless apps instead lean on real `await`/microtask flushing or `TestBed.tick()`).

### 4.3 How Marble Syntax Encodes Virtual Time

- `TestScheduler.run(callback)` swaps in a `VirtualTimeScheduler`-derived scheduler and monkey-patches operators that use `AsyncScheduler` (like `debounceTime`, `delay`, `throttleTime`) to schedule against virtual frames instead of real wall-clock time.
- Each character in a marble string represents one virtual "frame" (by default, a fixed frame size, often treated as 1ms-equivalent, though the actual duration is whatever the operator's time argument specifies — the *frame index* is what matters, not real elapsed ms).
- Inside `scheduler.run()`, `cold()`/`hot()` build `ColdObservable`/`HotObservable` test doubles that schedule their marble-described emissions onto the virtual scheduler's frame queue at parse time.
- After the callback runs (queuing subscriptions and expectations), `TestScheduler` **synchronously flushes** the virtual clock: it processes the queued frame events in order, feeding scheduled notifications to subscribers exactly as if real time had elapsed, then compares the actual recorded notifications on each `expectObservable` result against the parsed expected marble string — deep instant, synchronous assertions, immune to real timer flakiness.

## 5. Edge Cases & Gotchas

- **Forgetting `detectChanges()` → false negatives (or false positives)**: a test that mutates component state and asserts against the DOM without a following `detectChanges()` call is actually asserting against **stale/pre-render** DOM. Worse, a test written to check "the error message appears" might pass **even after the feature is broken**, if it never called `detectChanges()` at all (asserting an *absence* that was true both before and after implementation) — silent false-positive coverage.
- **`async`/`waitForAsync` vs `fakeAsync`/`tick()` confusion**:
  - `waitForAsync` (formerly `async`, renamed to avoid clashing with the JS keyword) runs the test body inside Angular's async test zone and lets you use real Promises/`fixture.whenStable()`; it does **not** let you control/advance time — real timers still fire on their own schedule (in real time, though usually near-instant for microtasks).
  - `fakeAsync` + `tick(ms)` runs inside a *patched* zone where `setTimeout`/`setInterval`/Promise microtasks are all captured; nothing fires until you call `tick(ms)` (advance virtual time by ms) or `flush()` (run all pending timers to completion). Mixing them (`fakeAsync` around code that internally uses raw XHR outside zone patching, or forgetting to `tick()` past a `debounceTime`) is one of the most common Angular testing bugs — a test **hangs or silently never resolves** the assertion, or throws "N timer(s) still in the queue" at `fakeAsync` teardown if you forget to `flush()`/`tick()` enough.
  - Combining `fakeAsync` with real (non-virtual) RxJS `TestScheduler` marble tests in the same spec is redundant/conflicting — pick one virtual-time mechanism per test.
- **Effects not flushing**: writing to a signal inside a test and immediately asserting an `effect()`'s side-effect (e.g., a `localStorage.setItem` call) without a subsequent `detectChanges()`/`await` will fail because effects are scheduled, not synchronous. This surprises engineers coming from `ngOnChanges`/`RxJS` `Subject.next()` mental models where subscribers run synchronously.
- **`NO_ERRORS_SCHEMA` masking real bugs**: overusing this schema to silence "unknown element" errors from unregistered child components can hide genuine template typos (misspelled input bindings on your **own** component under test), not just child component markup — audit its use.
- **`httpMock.verify()` omitted**: without calling `verify()` in `afterEach`, a component that accidentally fires an extra unexpected HTTP call (e.g., double-subscribing an observable) goes undetected — the test suite stays green while production makes duplicate network calls.
- **Testing `OnPush` + `ChangeDetectorRef.markForCheck()` interplay**: a component that mutates internal state from a non-Angular-triggered source (e.g. a WebSocket callback, a third-party library callback outside `NgZone`) and calls `markForCheck()` needs the **parent fixture's** `detectChanges()` to actually walk down to it; calling `markForCheck()` alone only flags it dirty, it does not itself trigger a CD pass.
- **Flaky E2E tests from timing assumptions**: hardcoded `cy.wait(1000)` / `page.waitForTimeout(1000)` calls instead of waiting on a specific condition (element visible, network response settled, URL change) are the number one cause of intermittent E2E failures under CI load variance. Both Cypress and Playwright provide auto-retrying assertions/actionability checks (`cy.get(...).should(...)`, `expect(locator).toBeVisible()`) specifically to avoid arbitrary sleeps — relying on fixed sleeps is a code-review red flag.
- **Shared mutable test fixtures across `it()` blocks**: forgetting that `TestBed` resets between tests but a module-level `let` variable holding e.g. an array pushed to in one test can leak state into the next if not reset in `beforeEach`.
- **Marble test frame-size mismatches**: writing `debounceTime(300)` in source code but a marble diagram whose dash-count doesn't correspond to 300 (the *default* virtual frame unit is decoupled from literal milliseconds unless you keep them consistent) — always pass the same `scheduler` instance into the operator under test (`debounceTime(300, scheduler)`) so the virtual scheduler, not `asyncScheduler`, is actually used; forgetting the second argument silently falls back to real-time async scheduling inside a synchronous test, causing it to hang or immediately emit at the wrong frame.

## 6. Interview Questions & Answers

**Q1. What is the difference between `TestBed.createComponent()` and just `new`-ing a component class directly in a test?**
`new MyComponent()` bypasses Angular's dependency injection, template compilation, and change detection entirely — it only works for components with no constructor dependencies and no template to verify, essentially testing it as a plain class. `TestBed.createComponent()` goes through Angular's actual bootstrap machinery: it resolves the component's DI graph from the configured testing module, compiles/renders its real template, and returns a `ComponentFixture` giving you `nativeElement`, `debugElement`, and `detectChanges()` — necessary for testing anything involving bindings, DI, lifecycle hooks, or child components.

**Q2. Why does the Angular team recommend `imports: [MyStandaloneComponent]` instead of `declarations` in `TestBed.configureTestingModule` for a standalone component?**
*Interviewer intent: checks whether the candidate actually understands the standalone API's mental model rather than pattern-matching old Karma boilerplate.*
Standalone components/directives/pipes are, by definition, self-contained units that declare their own dependencies via their `imports` array in the `@Component` decorator — they are never `declared` by an `NgModule`. `TestBed`'s testing module for a standalone component therefore treats it the same way any consumer would: as an importable unit. Putting it in `declarations` throws an error at test setup ("... is standalone and cannot be declared") because Ivy explicitly forbids that mismatch to prevent a whole class of DI/module confusion bugs.

**Q3. A test asserts DOM text right after mutating a component property but the assertion fails, showing the old value. What's the most likely cause and fix?**
The test forgot to call `fixture.detectChanges()` after the mutation. Angular's `ComponentFixture` does not auto-run change detection between statements by default (`autoDetectChanges` is off) so the rendered DOM only reflects the state as of the last explicit `detectChanges()` call. Fix: call `fixture.detectChanges()` again after the mutation, then assert.

**Q4. Why might you deliberately choose "shallow" testing (stubbing child components) over "deep" (rendering real children) for a given parent component test?**
Shallow testing isolates the parent's own template-binding logic and event wiring from the correctness of its children, so the test doesn't break when an unrelated child component's internals change, and doesn't need every transitive dependency of every child satisfied in the testing module (faster setup, faster execution, fewer false failures from unrelated regressions). The trade-off is reduced confidence that the components actually work correctly *together* — that's what integration/E2E tests are for.

**Q5. What's the internal reason `fixture.detectChanges()` in a unit test doesn't behave exactly like the automatic change detection you get in a running app?**
*Interviewer intent: probes understanding of Zone.js/NgZone's role versus TestBed's deliberately manual test harness — a common area where candidates have surface familiarity but not internal understanding.*
In a running app, `NgZone` patches async APIs via Zone.js and fires `ApplicationRef.tick()` automatically whenever the zone becomes stable (`onMicrotaskEmpty`), so change detection runs continuously in response to real async events. `TestBed`'s `ComponentFixture` deliberately opts out of that automatic hook by default (`autoDetectChanges(false)`) so tests can assert intermediate/pre-render state deterministically; the test author must call `detectChanges()` explicitly to drive a CD pass. Calling `fixture.autoDetectChanges(true)` re-subscribes to the same zone-stable signal used in production, trading determinism for closer-to-production automatic behavior. Under zoneless change detection, the underlying trigger is the signal-based scheduler rather than `NgZone`, but the same manual-vs-auto choice on `ComponentFixture` still applies.

**Q6. How do you test a component whose only "input change" is a mutation of a nested property inside an object `@Input`, under `OnPush`?**
Mutating a nested property without replacing the object reference does not, by itself, mark the OnPush view dirty (`ngOnChanges` won't fire either, since the reference is unchanged) — this holds true in production. In a unit test where the component under test is the **fixture root**, `fixture.detectChanges()` still forces a fresh check of that component regardless of strategy, so the DOM *will* reflect the mutation once you call it. The gotcha shows up when testing a **parent-child** relationship: if the child is OnPush and you mutate the object passed from the parent without replacing the reference, calling `detectChanges()` on the parent fixture will not cause the child to see a "changed" input via `ngOnChanges`, though the child's own template read of the mutated nested property (if not memoized) may still reflect the new value once its view is checked — so tests should assert based on what the framework actually triggers (identity-based `ngOnChanges`/input change detection) not general "did the DOM update" assumptions.

**Q7. Why does an `effect()` not run synchronously immediately after the signal it depends on is set, inside a test?**
`effect()` runs on Angular's internal reactive scheduler, which batches and defers execution (similar in spirit to microtask coalescing) rather than firing synchronously on every dependency write — this avoids redundant re-runs when multiple signals are updated in the same tick. In tests, you need to explicitly flush that scheduler — via `fixture.detectChanges()` (which flushes pending effects as part of its CD cycle when effects are registered in a component's injection context), an `await` on a microtask, or `TestBed.tick()` — before asserting effect-driven side effects.

**Q8. Explain what a marble string like `'-a-b-(c|)'` means, and what `cold()` vs `hot()` represent.**
Each character is one virtual frame. `-` is an empty frame (no emission). `a` and `b` are `next` notifications emitted at their respective frame positions, mapped to real values via the values object passed to `expectObservable`/`cold`. `(c|)` groups a `next('c')` and a `complete` notification into the same frame (parentheses mean "simultaneous events"). `cold()` observables start their timeline from the moment they are *subscribed to* in the test (like a fresh HTTP call or a `Subject` unrelated to prior activity) — each subscriber gets its own independent execution starting at frame 0 relative to its own subscription. `hot()` observables have a shared, ongoing timeline from frame 0 of the test regardless of when a given subscriber subscribes (like a shared `Subject` or UI event stream) — a late subscriber can miss earlier emissions, mirroring how a real hot observable (e.g. a click stream) behaves.

**Q9. How does `HttpTestingController.verify()` catch bugs that individual `expectOne()` assertions wouldn't?**
`expectOne()` only asserts about the specific request URL/method you explicitly checked for; it says nothing about *other* requests that might have also fired. `verify()`, typically called in `afterEach`, asserts that there are no outstanding (unflushed) requests and — critically — throws if there were requests made that no `expectOne`/`match` call accounted for, catching accidental duplicate calls (e.g., a component subscribing to an HTTP observable twice, once in `ngOnInit` and once from a stale subscription) that would otherwise pass silently.

**Q10. Why would you use a CDK `ComponentHarness` instead of directly querying `.mat-mdc-*` classes or the raw DOM in a Material-based component test?**
*Interviewer intent: separates candidates who've only used Material from those who understand testing architecture / API contract stability, and can articulate the risk of relying on library internals.*
Material's internal DOM structure and CSS class names are private implementation details that can (and do) change between Material versions (e.g., MDC-based rewrite changed nearly all class names from `mat-*` to `mat-mdc-*`). A `ComponentHarness` exposes a stable, semantic API (`.click()`, `.getValue()`, `.isDisabled()`) that Material's own team maintains and updates in lockstep with internal DOM changes, so tests written against the harness survive library upgrades without modification. Harnesses also work transparently across execution environments (`TestbedHarnessEnvironment` for unit tests, a Protractor/E2E harness environment for E2E), letting the same harness-based test logic be reused or ported between layers.

**Q11. What's the practical difference between `fakeAsync`/`tick()` and RxJS marble testing (`TestScheduler`) — when would you reach for each?**
`fakeAsync` patches the zone so that *all* async APIs used anywhere in the code under test (timers, promises, XHR-if-configured) are captured and only progressed when you call `tick(ms)`/`flush()` — it is broad-spectrum and works regardless of what async mechanism is used, making it well suited for testing component/service code that mixes `setTimeout`, Promises, and Observables together, especially when integrated with Angular's own async lifecycle (e.g., testing a component that debounces via `setTimeout`). `TestScheduler` marble testing is RxJS-specific and only reliably virtualizes time for operators that accept an explicit `SchedulerLike` argument (`debounceTime(ms, scheduler)`, `delay(ms, scheduler)`); it is the right tool when unit testing a **pure RxJS operator pipeline/stream** in isolation (e.g., a service method returning an `Observable` built from several operators), because it gives you precise, readable frame-by-frame diagrams and doesn't need Angular's zone machinery at all — you could use it in a plain non-Angular RxJS library's tests too.

**Q12. Why is Karma being phased out by the Angular team, and what replaces it?**
Karma requires launching and orchestrating a real (or headless) browser process for every test run, which is comparatively slow to boot and awkward to run in constrained CI/sandboxed environments, and Karma itself has had reduced maintenance activity upstream. The Angular CLI has moved toward build-tool-native test runners: an esbuild/Vite-based **Web Test Runner** integration (still exercising a real browser, but with faster incremental builds) and native support for **Jest** (running in Node with jsdom, offering much faster startup and better watch-mode ergonomics, at the cost of jsdom's imperfect DOM/CSSOM fidelity versus a real browser). Recent Angular CLI versions no longer scaffold Karma by default for new projects.

**Q13. How would you unit test a service method that returns `Observable<T>` built via `switchMap` over an HTTP call, ensuring only the latest request's result is used when the trigger fires rapidly?**
Two viable approaches: (a) use `HttpTestingController` and drive the trigger subject/method twice in quick succession, then assert via `httpMock.expectOne`/`match` that both underlying HTTP calls were made (`switchMap` cancels the *subscription*, not necessarily the already-sent HTTP request unless using `takeUntil`/cancellation-aware backend) and that only the last response resolves to the final emitted value — flush the second request first, assert on the observable's result, then flush/verify the first is unsubscribed rather than incorrectly contributing a value; or (b) use RxJS marble testing with `cold()` observables standing in for the two "requests" and assert the composed pipeline's output diagram shows only the later inner observable's values, which more precisely and deterministically demonstrates the `switchMap` cancellation semantics without real HTTP mocking overhead. In an interview, mentioning both and explaining the trade-off (marble tests isolate operator logic; `HttpTestingController` tests the actual service+HTTP integration) demonstrates the breadth expected at this level.

**Q14. What causes an E2E test to be "flaky," and what's the fix beyond "just add a wait"?**
Flakiness usually stems from an assumption that a UI state is ready at a fixed point in (real) time — network latency, animation duration, or CI machine load variance — rather than the actual asynchronous condition the test depends on. The fix is to assert on the *condition itself*: wait for an element to become visible/enabled, wait for a specific network response to resolve (`cy.intercept().as('alias'); cy.wait('@alias')`, or Playwright's `page.waitForResponse()`), or wait for a URL/route change, rather than a fixed `wait(ms)`. Both Cypress and Playwright build in automatic retrying assertions and actionability checks specifically to eliminate the need for manual sleeps; persistent flakiness after using those usually points to a genuine race condition or non-deterministic backend state in the app itself, which is valuable signal, not just a test-authoring problem to suppress.

**Q15. Describe the test pyramid trade-off in the context of a large Angular enterprise app — why not just write E2E tests for everything, since they give the most confidence?**
*Interviewer intent: tests whether the candidate can reason about engineering economics/CI cost, not just tool mechanics — a hallmark staff-level question.*
E2E tests are expensive along every axis: slow to run (real browser boot + network + animations), expensive to maintain (brittle to UI/markup changes, timing-sensitive, need a running backend or elaborate mocking), and slow to debug when they fail (less granular failure information than a unit test pinpointing a single function). If E2E were the only layer, CI feedback loops would stretch from minutes to hours as the app grows, and a single flaky test can block an entire pipeline. Unit and component tests catch the vast majority of logic regressions in milliseconds with precise failure localization, at low authoring/maintenance cost, but say nothing about whether the pieces actually integrate correctly in a real browser/user journey — which is exactly the gap a **small number of** high-value E2E tests (critical user journeys: login, checkout, core workflow) exist to cover. The pyramid shape is precisely this cost-weighted allocation: broad, fast, cheap coverage at the bottom; narrow, slow, expensive but high-fidelity coverage at the top.

## 7. Quick Revision Cheat Sheet

- **Test pyramid**: many unit tests → some integration/component tests (TestBed) → few E2E tests (Cypress/Playwright).
- **Karma deprecated**; CLI moving to Web Test Runner (real browser, esbuild/Vite-native) and native **Jest** support (Node + jsdom, faster, snapshot testing, less DOM fidelity).
- **TestBed**: `configureTestingModule({ imports: [StandaloneComp], providers: [...] })` — standalone components go in `imports`, never `declarations`.
- **`fixture.detectChanges()`** is manual by default (`autoDetectChanges(false)`) — call it again after every state mutation you want reflected in the DOM; forgetting it is the #1 cause of false test results.
- **Shallow vs deep**: stub children (`NO_ERRORS_SCHEMA` or explicit fake components) to isolate the parent; use real children for integration confidence.
- **OnPush**: `detectChanges()` on the fixture root still forces a check regardless of strategy; the nuance only bites when testing parent→child `@Input` propagation via mutated (non-replaced) references.
- **Signals**: synchronous reads/writes, no async ceremony. **`computed()`**: pull-based/memoized, correct on next read. **`effect()`**: scheduled, not synchronous — flush via `detectChanges()`, `TestBed.tick()`, or awaiting a microtask.
- **Marble testing** (`TestScheduler`): `-` = frame, letter = next value, `|` = complete, `#` = error, `(xy|)` = simultaneous events. Always pass the scheduler explicitly into time-based operators (`debounceTime(ms, scheduler)`).
- **`fakeAsync`/`tick()`**: broad-spectrum virtual time for timers/promises/zone-patched async; **`waitForAsync`**: real async completion via `whenStable()`, no manual time control.
- **HttpTestingController**: `provideHttpClient()` + `provideHttpClientTesting()`, `expectOne(url)` → `req.flush(data)` / `req.error(...)`, always `httpMock.verify()` in `afterEach` to catch unexpected/duplicate calls.
- **CDK `ComponentHarness`**: stable semantic API insulating tests from Material's internal DOM/class churn; `TestbedHarnessEnvironment.loader(fixture)` for unit tests.
- **Cypress vs Playwright**: Cypress = in-browser architecture, mature ecosystem, time-travel debugger; Playwright = true multi-browser (Chromium/Firefox/WebKit), fast native parallelism, trace viewer. Both replaced deprecated Protractor.
- **Flaky E2E fix**: assert on real conditions (element visible, network response resolved, URL changed) via auto-retrying assertions — never hardcode `wait(ms)`.
